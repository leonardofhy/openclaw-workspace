## 1. Core Taxonomy: Noising vs Denoising

The most important insight in the paper. These are NOT symmetric:

| Direction | Name | What it measures |
|-----------|------|-----------------|
| clean → corrupt | **Denoising** | Which activations are **sufficient** to restore behaviour |
| corrupt → clean | **Noising** | Which activations are **necessary** to maintain behaviour |

**AND vs OR gate logic (critical for audio):**

- AND circuit (A AND B → C): **Noising finds all; Denoising finds only C** (because denoising A leaves B corrupted)
- OR circuit (A OR B → C): **Denoising finds all; Noising finds only C** (because noising A leaves B clean)

**Audio implication**: AudioSAE showed speech is DISTRIBUTED — 1000s of features needed to erase a concept. This suggests REDUNDANT (OR-gate) structure. **Therefore: denoising is better for audio patching experiments** (denoising finds all branches of OR-gate). This is the opposite of what most text MI researchers default to (noising).

This is a **concrete design decision for Track 1 (Audio Causal Benchmark)** and Gap #18: use denoising direction for audio feature exploration.

## 2. Granularity Ladder

Recommended progression for patching:
1. **Residual stream patching** (coarsest, fastest) → which layers?
2. **Component patching** (MLP/attention head outputs) → which components?
3. **Path patching** → do A→B directly interact, or via C?
4. **SAE feature patching** → individual interpretable features (highest granularity)

**For audio**: Starting at SAE feature level (Gap #22 proposal) is the RIGHT endpoint, but should be validated by first confirming layer-level results (residual stream sweep → layer 6-7 transition zone already identified by AudioSAE + Beyond Transcription).

**Path patching** distinction: tests whether two components interact **directly** (composition) vs **via mediation**. Important for understanding connector geometry (Gap #18): does the phonological vector pass THROUGH the connector directly, or is it re-encoded at the connector?

## 3. Corruption Design — Most Underrated Pitfall

Patching only traces differences between clean and corrupt prompt. Must know:
- What does the corruption change?
- What does it NOT change?

**Examples from paper (IOI task):**
- Replace "Mary" with random name → changes person identity only
- Replace "John and Mary" → changes both actors
- Scramble word order → changes many things at once (bad for exploration)

**Audio translation:** 
- Replacing one phoneme (e.g., /b/ → /p/) = minimal pair = ideal for phonological testing (Gap #18)
- Adding Gaussian noise to waveform = changes MANY features at once (less controlled)
- AudioSAE uses "non-speech vs speech" as clean/corrupt = very wide distribution (good for hallucination steering, bad for localized phoneme patching)

**Recommendation for Track 1 design:** Use **minimal pairs** (phonological, prosodic, semantic contrast) as the standard corruption protocol. Wide corruptions are fine for confirmatory patching but bad for exploration.

## 4. Metric Choice — Pitfalls

Key advice: implement multiple metrics and check agreement.

| Metric | Strength | Weakness |
|--------|----------|----------|
| **Logit difference** (clean vs corrupt logits) | Linear in residual stream; controllable | Requires knowing correct/incorrect answer token |
| **Log probability** | Natural information metric | Nonlinear → hard to attribute |
| **Probability difference** | Intuitive | Saturates near 0/1, misleading near boundaries |
| **KL divergence** | Full distribution; catches negative components | Tracks irrelevant variance too |
| **Internal metrics** (attention, SAE feature activation, probe) | Circuit-level; testable | Requires prior hypotheses |

**Audio-specific pitfall:** WER (Word Error Rate) is a **sequence-level metric** — patching changes may have large WER impact from single token errors, or small WER impact from many distributed changes. This maps to Research Question #3 in goals.md.

**Best metric for audio:** Use **per-token log probability of the correct phoneme/word** (pinpoint) + **WER as secondary** (global). For continuous audio outputs, use **feature activation of target SAE feature** as internal metric.

## 5. Interpretation Pitfalls

### No Minimality
A found "circuit" may not be the **smallest** sufficient circuit. Standard patching never claims minimality — only sufficiency or necessity.

**Audio implication**: The "layer 6-7 transition zone" finding from AudioSAE + Beyond Transcription is a **necessity** finding (noising direction in both papers). It doesn't mean these are the only layers that matter.

### Backup Behaviour (Hydra Effect)
When you ablate/noise a key component, a backup head/feature activates to compensate. The original component's effect appears weakened (by ~0.7x in IOI study). 

**Audio implication**: Speech representations are DISTRIBUTED → backup behaviour is likely STRONG in audio models. This explains why AudioSAE's unlearning curves are slow: each feature erased triggers backup features. **For AudioSAEBench Gap #22**: when testing causal utility of stable features, must account for Hydra effect → use aggregate patching of top-k consistent features, not single feature.

### Negative Components
Some circuit components **suppress** the correct answer (anti-features). Patching these improves performance. Makes it hard to tell "I've found a good circuit" from "I've found some positive components but missed all negative ones."

**Audio safety implication (Track 5)**: Safety-suppressing attention heads might look like "negative components" in patching — removing them IMPROVES harmful output. This is exactly the Feng et al. mechanism (Gap #20) → patching identifies suppression heads.

## 6. Scoping Limitation

> "Activation patching is always based on prompt distributions, and does not make statements for model behaviour outside these specific distributions."

**Critical for Leo**: Any causal claims from audio patching apply ONLY to the specific clean/corrupt pairs used. The "layer 6-7 is the transition zone" finding from Beyond Transcription holds for THEIR corruption (clean speech vs white noise reference). For phonological minimal pairs (Gap #18), the transition zone might be different.

**Design rule for Track 1**: Define **multiple corruption distributions** (at least 3: waveform-level, phonological, semantic) and sweep each separately. Report which layers are necessary/sufficient per distribution type.

## 8. New Cheat Sheet Entry

Key formulas to remember:
- **Normalized patching effect** = (metric_patched - metric_corrupt) / (metric_clean - metric_corrupt)
  - 0 = no recovery; 1 = full recovery
  - >1 means overshoot; <0 means interference
- **Logit difference** for binary tasks = logit(correct) - logit(incorrect) — best default metric
- **Attribution patching (AtP)** = fast approximation using gradients × activation differences (order of magnitude faster, good for large model sweeps)

## 9. Open Questions / Implications

**For Gap #18 design (Priority 0):**
- Which corruption to use for phonological vector test? 
  - Option A: Choi et al. minimal pair stimuli (e.g., /b/ vs /p/ words) → narrow, controlled
  - Option B: Generate pairs with TTS with different voicing → flexible but model-dependent
  - **Recommendation: Option A (principled, existing validated stimuli)**

**For AudioSAEBench causal metric design:**
- Hydra effect means single-feature patching has weakened signal. Paper confirms: backup effect = ~0.7x reduction.
  - **Design implication**: Use TOP-K aggregate patching in AudioSAEBench causal utility tests; report k-saturation curve
  - This distinguishes "causally strong single feature" (no backup → full effect) from "causally weak feature" (backup activates → 0.7x effect)

**For Track 1 standardization:**
- The AND/OR gate classification of audio tasks is UNKNOWN — nobody has done this systematically for audio
  - **New mini-question**: Is phoneme detection an AND-gate (requires both phoneme-level AND sentence-level context) or OR-gate (any route suffices)?
  - If AND-gate: noising finds all components (good for exploration)
  - If OR-gate: denoising finds all components (requires knowing structure first)
  - → Start with noising sweep (safer for unknown structure) → classify as AND/OR from results
