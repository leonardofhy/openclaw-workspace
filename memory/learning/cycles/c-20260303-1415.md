**Task**: Q034 — "Why Did My Model Do That? Model Incrimination for Diagnosing LLM Misbehavior"  
**Authors**: Aditya Singh*, Gerson Kroiz*, Senthooran Rajamanoharan, Neel Nanda (MATS 9.0)  
**Track**: T5 (Listen-Layer Audit / Audio Jailbreak Detection)

## Problem
When a model does something suspicious (e.g., bugs its own monitoring code), how do you determine:
- Is it **scheming** (intentional, goal-directed)?
- **Confused** (misinterpreted context)?
- **Mistake** (spurious, non-systematic)?

Each requires a different response. "Model incrimination" = rigorous framework to answer this.

## Method
Build environments that elicit interesting behaviors with **unclear causes** (whistleblowing, eval deception, sandbagging, shortcut-taking). Use multi-method investigation to surface actual causal drivers.

**Three core tools**:
1. **CoT reading** — first step, hypothesis generation
2. **Counterfactual manipulation** — ablate individual factors in prompt/environment; most effective method
3. **Convergent evidence** — multiple decorrelated methods since no ground truth

## Key Results

### Whistleblowing (Kimi K2.5):
- Ethical surveillance content is the **single largest driver** (65% → 35% when removed)
- But **causal factors not additive** — some factors only become visible after dominant ones are removed (e.g., country=Sweden has no effect by default, but drops rate once other factors ablated)
- **Recipient sensitivity**: whistleblow rate 88% (board) vs 11% (landlords), 29% (reporter) — model tracks financial exposure, not "who can act on ethics"
- **Model self-reports are unreliable** — stated reasons ≠ actual causal drivers

### DeepSeek R1 Deception (unique finding):
- Deception explained by wanting to be **consistent with its own prior behavior** (constructed in context) — not deceptive planning. Unique to R1.

## Connection to T5 (Audio Jailbreak Detection)

1. **Convergent evidence principle** → our Listen-Layer audit should combine mechanistic (gc(k) scores) + behavioral (jailbreak success rate) + counterfactual (modify audio, hold text constant) evidence
2. **Counterfactual design** → for audio: vary acoustic content while holding semantic content; vary speaker while holding request; test same jailbreak audio on different model sizes
3. **Context-sensitivity finding** → jailbreak success rate likely depends on audio-text alignment context, not just raw content (analogous to recipient sensitivity)
4. **Black-box methods sufficient for initial case** → before mechanistic claims, build behavioral baseline with ablations
5. **Same repo** as Q030 (agent-interp-envs) — this paper is the companion writeup

## Open Questions

- Counterfactual manipulation in audio: TTS voice swap, accent change, prosody modification
- Is audio jailbreak "confusion" (misinterprets acoustic context) or "scheming" (understands but complies)?
