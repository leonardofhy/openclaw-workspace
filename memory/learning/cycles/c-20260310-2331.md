### Idea #3: SAEBench "Grounding Sensitivity" as an Audio-Native Metric
**Elements combined:** SAEBench (8-metric multi-category, proxy ≠ quality) × gc(k) grounding coefficient (T3) × AudioSAE
**Research question:** Can "grounding sensitivity" — how much activating a SAE feature shifts audio-vs-text reliance — become a 9th SAEBench category for audio models?
**Approach:** For each AudioSAE feature, measure Δgc(k) when the feature is clamped to high vs zero. Features with high ΔGS = grounding-critical; features with low ΔGS = text-prediction shortcuts. This is a new evaluation axis not in SAEBench, complementary to the existing 8 categories.
**Novelty:** 4/5 — Grounding Sensitivity named but not formally defined/tested in any benchmark
**Feasibility:** 5/5 — Pure CPU computation on existing AudioSAE checkpoints + gc_eval harness
**Verdict:** ✅ novelty+feas = 9 → KEEP (HIGH PRIORITY — builds directly on current infrastructure)

### Idea #18: "Interpret Then Defend" Pipeline for Audio Jailbreaks
**Elements combined:** T5 safety × SAE adversarial detector (Q041 done) × Geiger causal abstraction
**Research question:** Can we build an "interpret then defend" pipeline: first use SAE to attribute which features are activated by a jailbreak, then causally intervene on exactly those features?
**Approach:** Step 1: SAE attribution identifies top-k features with high ΔActivation (clean vs adversarial). Step 2: Causal patching from clean→corrupt replaces only those k features. Step 3: Measure robustness + collateral damage vs SPIRIT baseline (which patches whole neurons).
**Key novelty:** Feature-level patching = finer scalpel than neuron-level; Geiger's causal abstraction justifies why this is the right level of abstraction
**Novelty:** 4/5 — SPIRIT does neurons; nobody does SAE-feature-level patching for defense
**Feasibility:** 5/5 — Q041 already has SAE adversarial detector; feature patching = small extension
**Verdict:** ✅ novelty+feas = 9 → KEEP (HIGH PRIORITY — extends done work by 1 step)
