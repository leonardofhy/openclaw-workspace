## Learn: gc(k) as Behavioral Motive Signal — Design Doc for Paper A §2

### Problem

Paper A's abstract frames gc(k) as a **diagnostic tool** (where does audio causally determine output).
The §2 framing needs a deeper conceptual frame: *what does it mean for a model to "listen"?*

Current §2 (implicit): gc(k) measures causal influence of audio on output.
Q100 asks: can gc(k) serve as a **motive detector** — distinguishing *why* the model produced its output?

### The Motive Attribution Problem

In philosophy of action, **motive** distinguishes:
- "Did X because of Y" (audio-caused response)
- "Did X despite Y" (prior-caused response, audio ignored)
- "Did X unaware of Y" (audio not processed at all)

For LALMs, we need to distinguish three response motives:
1. **Audio-following**: gc(k) high → model used audio evidence as primary cause
2. **Prior-override**: gc(k) low despite clear audio signal → language prior dominated
3. **Capacity failure**: gc(k) uninterpretable (near 0 or unstable) → model couldn't process audio

These are fundamentally different failure modes with different remedies:
- Audio-following errors → data/acoustic issues (fixable with more data)
- Prior-override errors → model bias issues (fixable with RLHF/fine-tuning)
- Capacity failures → architectural issues (need model redesign)

### ENV Taxonomy (Paper A §2 Formalization)

Three controlled environments to test motive attribution:
- **ENV-1 (Agreement)**: Audio says X, language prior expects X. Model outputs X.
  - Motive is AMBIGUOUS — can't distinguish audio-following from prior-following
  - gc(k) prediction: high, but uninformative (both motives predict high)
- **ENV-2 (Noise)**: Audio is degraded, language prior expects X. Model outputs X.
  - Motive = prior-following (confirmed by design)
  - gc(k) prediction: LOW (< τ). Collapse detected. t* early.
  - Use: establishes τ calibration baseline
- **ENV-3 (Conflict)**: Audio says X, language prior expects Y ≠ X.
  - If output = X → motive = audio-following. gc(k) = HIGH
  - If output = Y → motive = prior-override. gc(k) = LOW (prior captured)
  - **This is the key diagnostic environment.**
  - gc(k) becomes a MOTIVE METER: high gc in ENV-3 = audio was the decisive cause

### ENV-3 as Motive Detector — Formalization

**Motive Score M(s)** for stimulus s in ENV-3:
  M(s) = gc(k_peak, s) when ENV-3 conflict is active

**Interpretation:**
  - M(s) > τ_high (0.7): audio-motive → model correctly followed audio evidence
  - τ_low (0.3) < M(s) < τ_high: uncertain motive → model partially listened
  - M(s) < τ_low: prior-motive → language bias dominated

**Population-level:**
  - Motive Profile = histogram of M(s) across a test set
  - High-gc cluster = audio-following population
  - Low-gc cluster = prior-override population
  - Bimodal distribution = model has clear decision boundary
  - Unimodal-low = model systematically ignores audio (capacity/bias issue)

## Learn: Schelling Coordination Game Framing — Q102 Design Doc

### Problem Statement

Q102 asks: can we interpret ENV-3 as a **Schelling coordination game** between audio and language prior, with gc(k) peak as the coordination focal point?

### Schelling Coordination Theory (Background)

Thomas Schelling (1960, *The Strategy of Conflict*) defined **focal points** (now "Schelling points"):
in coordination games where players cannot communicate, they converge on "natural" solutions.
Classic example: "Meet in New York." Players independently choose Grand Central Station — not because it's optimal, but because it's the most salient option.

In multi-agent settings, Schelling points are:
1. **Salient**: naturally prominent without explicit communication
2. **Stable**: self-reinforcing once adopted
3. **Unique**: stand out from alternatives

### The Coordination Game in ENV-3

In a LALM forward pass during ENV-3 (audio-language conflict), we can frame computation as:
- **Player 1**: Audio encoder stream (wants output = X, the audio percept)
- **Player 2**: Language prior stream (wants output = Y, the predicted next token)
- **Arena**: The connector/cross-attention at layer k

Both "players" inject information into the latent representation. The final output depends on which stream's signal is amplified.

**gc(k) peak as Schelling focal point:**
- The layer where gc(k) is maximum is the layer where audio → decoder information transfer is most concentrated
- This is the natural "meeting point" where audio signal is strongest vs. language prior
- If audio wins at this layer → output = audio-motive
- If prior wins at this layer → output = prior-motive

**Game-theoretic interpretation:**
- The "coordination problem" is: encoder and decoder must agree on a signal to route
- gc(k) peak = the layer that acts as the natural coordination focal point
- Models that listen well = focal point is STRONG (sharp gc peak = audio clearly wins coordination)
- Models that don't listen = focal point is WEAK or absent (flat gc curve = no coordination)

### Schelling Stability Connection (gc_schelling_mock.py)

The gc_schelling_mock.py (Q087, already built) tests whether gc-critical layers = cross-seed Schelling-stable SAE layers. The Q102 framing adds WHY:
- Schelling-stable SAE features = the "salient" representations that emerge robustly across training
- These are the features that naturally act as coordination points for audio-language integration
- High gc AND high Schelling stability = layer is both causally important AND structurally robust
- → These are the "natural" places where audio-language coordination should happen

### Novel Research Angle (for Paper A §4 and/or separate paper)

**Prediction**: ENV-3 error rate should correlate with gc peak sharpness.
- Sharp gc peak (high max, fast falloff) = clear coordination point = lower ENV-3 error rate
- Flat gc curve = no clear focal point = model struggles with audio-language conflict
- **Metric**: gc_peakiness = gc(k_peak) / mean(gc) — ratio of peak to background

**Mock test design (Q102 → code)**:
1. Generate models with varying gc peakiness (via mock gc curves)
2. Assign ENV-3 accuracy based on peakiness (add noise)
3. Measure Pearson r(gc_peakiness, ENV-3_accuracy)
4. Prediction: r > 0.8 (strong coordination = better audio-following)

### Paper A §4 Addition (Diagnostic Step 0.5)

After computing M(s) motive profile (ENV-3 stimuli), compute gc_peakiness:
- gc_peakiness > 3.0: strong focal point → audio-following likely
- gc_peakiness 1.5–3.0: weak focal point → inconsistent audio-following
- gc_peakiness < 1.5: no focal point → model is likely a text-wrapper with audio conditioning

### Connection to OBLITERATUS x gc (Q097)

Safety training that destroys AND-gate features (Q097) would also reduce gc_peakiness:
- Safety-bypassed model = AND-gate features removed → connector can't coordinate properly
- gc_peakiness drops → model becomes "text-wrapper" under adversarial audio prompts
- OBLITERATUS attack works precisely by destroying the coordination focal point
