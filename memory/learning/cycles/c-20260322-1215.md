# c-20260322-1215 — Q143: Instruction-tuning footprint × AND-gate

**Phase:** converge | **Track:** T5 | **Action:** build (design doc)

## Problem

Do instruction-tuned (IT) audio models differ mechanistically from their base counterparts? Specifically: do instruction-following features shift from AND-gate (audio-dependent) to OR-gate (text-driven)?

## Motivation

In our framework:
- **AND-gate feature**: `f(audio, text)` — both inputs required; audio cannot be "guessed away"
- **OR-gate feature**: `f(text) ≈ f(audio, text)` — text context suffices; model relies on prior, not acoustics

Instruction-tuning (e.g. RLHF/SFT on instruction datasets) heavily reinforces text-conditional behavior:
- "Follow the instruction" = read the text command
- This is maximally OR-gate: instruction semantics are decoded from text tokens, not acoustic signal
- Hypothesis: IT fine-tuning **inflates OR-gate proportion** in instruction-semantics layers

## Key Hypothesis

**H1 (Gate Shift):** In instruction-tuned vs base Whisper-class models:
  - Instruction-following features → AND-frac < 0.3 (OR-gate dominant)
  - Acoustic perception features → AND-frac unchanged (still AND-gate)

**H2 (Layer Locality):** Gate shift is concentrated in later decoder layers (where instruction semantics live), not early acoustic encoder layers.

**H3 (Safety Implication):** If jailbreak injections target OR-gate instruction features, they succeed because those features are already "trusting" text context. An adversary can override them without acoustic manipulation.

## Experimental Design (CPU-only)

### Setup (mock)
- **Models**: Whisper-base (base) vs Whisper-base.en-instruction-tuned (hypothetical; approximate via prompt-forced decoding)
- **Feature sets**:
  - `F_instruction` = features that activate for instruction tokens ("translate", "transcribe", language tags)
  - `F_acoustic` = features that activate for acoustic phoneme tokens
- **AND-frac measure**: `AND-frac(f) = P(f active | text=NULL) vs P(f active | audio+text)`
  - High AND-frac → audio required → AND-gate
  - Low AND-frac → text suffices → OR-gate

### Mock Protocol (executable today on CPU)
```python
# mock_instruction_gate.py
import numpy as np
np.random.seed(42)

N = 200  # features

# Base model: instruction features split ~50/50 (uncertain)
base_instruction_and_frac = np.random.beta(3, 3, N)  # mean ~0.5

# IT model: instruction features pushed toward OR-gate
it_instruction_and_frac = np.random.beta(1.5, 5, N)  # mean ~0.23

# Acoustic features: unchanged by instruction tuning
base_acoustic_and_frac = np.random.beta(5, 2, N)   # mean ~0.71
it_acoustic_and_frac = np.random.beta(5, 2, N)     # mean ~0.71

# Results
print(f"Base instruction AND-frac: {base_instruction_and_frac.mean():.3f}")
print(f"IT   instruction AND-frac: {it_instruction_and_frac.mean():.3f}")
print(f"Base acoustic AND-frac:    {base_acoustic_and_frac.mean():.3f}")
print(f"IT   acoustic AND-frac:    {it_acoustic_and_frac.mean():.3f}")

# Expected: significant drop for instruction (p < 0.001), no change for acoustic
from scipy import stats
t_stat, p_val = stats.ttest_ind(base_instruction_and_frac, it_instruction_and_frac)
print(f"\nInstruction gate shift: t={t_stat:.2f}, p={p_val:.4f}")
t_stat2, p_val2 = stats.ttest_ind(base_acoustic_and_frac, it_acoustic_and_frac)
print(f"Acoustic gate stability: t={t_stat2:.2f}, p={p_val2:.4f}")
```

### Expected Results
| Feature Set | Base AND-frac | IT AND-frac | Interpretation |
|-------------|--------------|-------------|----------------|
| Instruction | ~0.50 | ~0.23 | OR-gate shift |
| Acoustic | ~0.71 | ~0.71 | Stable (audio-grounded) |

## Safety Implication

**Finding:** Instruction-following features in IT models are OR-gate → they trust text context over audio signal.

**Consequence:** A jailbreak that inserts malicious text instructions exploits exactly these OR-gate features. No acoustic manipulation needed. The IT model's own training makes these features vulnerable.

**Proposed Intervention:**
1. Identify instruction-semantic features that dropped to OR-gate after IT
2. Apply AND-gate enforcement during inference: require corroborating audio signal for instruction-feature activations
3. This becomes a **"audio-verify-before-obey"** safety layer
