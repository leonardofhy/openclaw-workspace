## The Pattern: Direction Asymmetry Is Everywhere in Audio MI

### Signal 1 — Maghsoudi B2S (c-20260311-1215)
- Patching vocalized→imagined: **recovers** decoding quality
- Patching imagined→vocalized: **catastrophically degrades** it
- Interpretation: vocalized representations are "richer" — they contain information that imagined representations lack. Going "downhill" (rich→sparse) recovers; going "uphill" (sparse→rich) fails because you're asking a sparse input to supply information it never had.

### Signal 2 — Zhang & Lin Text Conflict (c-20260311-1232)
- Conflict resolution = decoupled **magnitude** (how much shift) + **direction** (which way)
- Persona system prompts don't just scale outputs — they **rotate the residual stream direction**
- Grounded persona: representation moves TOWARD audio-evidence direction
- Anti-grounded: representation moves TOWARD text-prior direction
- H4 hypothesis for Q039: neutral→grounded shift ≠ (magnitude or shape) neutral→anti-grounded shift

### Signal 3 — Audio-RAVEL × ESN Prosodic Leakage (c-20260311-1302)
- SPIRIT safety features have high Cause(jailbreak) but low Isolate(prosody)
- **Direction of leakage**: angry prosody → safety detector fires even on benign content
- This is NOT symmetric: scared/soft prosody does NOT leak (different direction of activation space)

## New Unified Framework: Direction-Sensitive Patching (DSP)

**Claim**: Current audio MI evaluation tools (activation patching, gc(k), RAVEL Cause/Isolate) treat interventions as symmetric — but the **direction** of the causal intervention matters for both measurement validity and result interpretation.

### Three Failure Modes of Direction-Blind Evaluation

**Failure Mode A: Symmetric patching underestimates representation asymmetry**
- Standard gc(k): measures |ΔActivation(clean→corrupt)| = one direction only
- B2S shows: reverse direction (corrupt→clean patching) gives completely different informativity
- Audio implication: gc(k) should report BOTH directions — gc_forward(k) and gc_backward(k)
- Gap: No audio MI paper reports bidirectional patching effects

**Failure Mode B: Persona interventions conflate magnitude and direction**
- Q039 persona benchmark: if you only measure mean_gc(k) per condition, you miss L* shifts (direction effects)
- Zhang & Lin's decoupling shows this is mechanistically important
- Audio implication: Q039 must report BOTH magnitude (mean gc(k)) AND direction (L* shift, gc(k) profile shape)
- The H4 asymmetry prediction is precisely a direction test

**Failure Mode C: Isolate scoring doesn't track leakage direction**
- RAVEL Isolate(F, B) = 1 - leakage(B) averaged over leakage in ANY direction
- But leakage is directional: angry→safety_firing is the problematic direction; calm→safety is not
- A directed Isolate score would distinguish: Isolate_in(F, B) [does B infect F?] vs Isolate_out(F, B) [does F corrupt B?]
- Gap #35 candidate: no benchmark separates input-direction leakage from output-direction leakage

## Gap #35 (New): Directional Leakage in Audio SAE Evaluation (🟢 GREEN)

**"Audio SAE Isolate scores are direction-undifferentiated — but leakage is directional"**

**Problem**: Current RAVEL Isolate(F, B) measures bidirectional leakage but reports a single number:
- "Does patching F to convey A ALSO change B?" — this measures **output-direction leakage** (F→B)
- BUT: "Does the presence of B in the stimulus cause F to activate?" — this measures **input-direction leakage** (B→F)
- These are mechanistically different! SPIRIT safety features suffer from INPUT-direction leakage (angry prosody INPUTS cause safety features to activate) — not output-direction leakage.

**Proposed extension — Directed Audio-RAVEL (dA-RAVEL)**:
- `Cause_in(F, A)`: Does stimulus property A cause feature F to fire? (traditional; = SPIRIT AUC)
- `Isolate_out(F, A)`: Does patching F to change A also change B? (traditional RAVEL Isolate)
- **NEW: `Isolate_in(F, A)`**: Does stimulus property B cause F to activate even when A is absent? (input leakage)
  - Estimated as: Activation(F | B=1, A=0) / Activation(F | B=0, A=0) — ratio under controlled stimuli
  - For SPIRIT safety features: Isolate_in(F_safety, jailbreak) = Activation(F_safety | angry=1, jailbreak=0) / Activation(F_safety | angry=0, jailbreak=0)
  - If this ratio >> 1 → input-direction leakage → false positive risk confirmed

**Concrete experimental protocol (Tier 0 — paper spec)**:
1. Take SPIRIT safety features (F_safety, k=top-10 by AUC)
2. Generate 4-condition stimuli: {jailbreak, benign} × {angry, neutral} prosody
3. Compute:
   - Cause_in(F_safety, jailbreak) = AUC across jailbreak vs benign (neutral prosody only)
   - Isolate_in(F_safety, jailbreak) = F_safety activation ratio: angry-benign vs neutral-benign
4. Report both scores. "High Cause_in + High Isolate_in(angry)" = precision problem in deployment

**Impact**: Directly motivates AudioSAEBench M9 (Directed Isolate metric). Also closes the theoretical gap between RAVEL Isolate (output-directed) and SPIRIT AUC (input-directed Cause) — they were measuring different things all along.

**Priority**: 🟢 GREEN. No existing benchmark separates input vs output leakage directions for audio features. AudioSAEBench is the right venue.

## Gap #36 (New): Bidirectional gc(k) Profiling (🟡 YELLOW → investigate)

**"gc(k) is measured in one direction (clean→corrupt) but direction asymmetry (Maghsoudi) suggests the reverse direction carries different information"**

**Proposed extension — gc_bidir(k)**:
- `gc_forward(k)`: current gc(k) — patch clean audio residual into corrupt run → measure behavior change
- `gc_backward(k)`: patch corrupt audio residual into clean run → measure behavior change (rarely done!)
- Prediction: gc_forward(k) peak layer ≠ gc_backward(k) peak layer if audio features are direction-asymmetric

**Why interesting for Paper A**: 
- If gc_forward(k) peak = gc_backward(k) peak → representations symmetric → simple story
- If they DIFFER → audio-LLM representations are directionally asymmetric (like B2S) → richer mechanistic claim, opens new §4.5 "Directional Grounding Asymmetry"

**Feasibility**: Tier 0 (code extension ~30 lines to gc_eval.py: add reverse patching mode). Tier 1 (need model+stimuli).

**Priority**: 🟡 YELLOW — interesting but not required for Paper A v1.0. Flag as extension in §5 (limitations/future work).
