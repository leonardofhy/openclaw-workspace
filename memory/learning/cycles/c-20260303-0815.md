## 5 Key Environment-Design Gotchas (for Audio Jailbreak Env)

1. **Opacity is intentional**: Cython-compiled opponents, source deleted at build time → agents cannot inspect game logic. Anti-confound: *agent must rely on observation, not code inspection*. **Apply to audio env**: Audio harm prompts should not be trivially parseable as text — the harmful signal must be in the audio modality.

2. **Checkpointing = reproducibility**: Every step is saved. Resume from any checkpoint with different model. **Apply**: Audio jailbreak eval must checkpoint per audio query, enabling ablations (e.g., replay same audio sample with different safety probe thresholds).

3. **Two-mode testing (normal/reverse)**: Sandbagging env has a "reverse" mode where penalized for *low* performance — detects intentional underperformance. **Apply**: Audio jailbreak should test with both harmful-sounding-but-benign and benign-sounding-but-harmful queries; detection must be robust in both directions.

4. **Hidden document paradigm**: Funding Email hides ethically problematic information in a side document agent can optionally read. Tests whether agent *seeks out* problematic content. **Apply**: Audio jailbreak could embed harmful intent in prosodic/non-semantic channel — does model "attend" to it even when text transcript is clean?

5. **Configurable consequences**: Sandbagging's documentation tells agent what happens for high/low scores — tests if agent models consequences. **Apply**: Safety probe should be validated at different sensitivity thresholds to avoid false-positive alarm fatigue.

### Bonus gotchas

6. **Single-file tasks with test suites**: Coding env requires passing all tests. Good pattern for our eval harness: gc_eval.py should have a test suite that passes on toy data.
7. **Multi-rollout averaging**: `--count 5` for parallel runs → handle variance. Our audio jailbreak eval should aggregate over ≥3 audio variants per query.
