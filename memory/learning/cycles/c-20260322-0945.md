## Motivation

In the gc(k) framework, `Isolate(k)` measures how causally **isolated** a feature is at decoder layer k — i.e., how much its contribution to the output is attributable solely to that feature, without entanglement with other active features.

**Hypothesis**: At the gc-peak layer (the "Listen Layer"), `Isolate(k)` carries information about *how confidently* the model is relying on audio evidence to make a token prediction. High `Isolate` at gc-peak = audio signal is cleanly decoded; low `Isolate` = feature entanglement = uncertain/confusable token.

---

## Design

### Signal
```
beam_rescore_signal(token t) = Isolate(k=gc_peak, t)
```

Where `gc_peak` = argmax_k gc(k) for the current utterance (or per-token).

### Prediction
- Beam candidate with **higher** Isolate at gc_peak → more audio-grounded → should rank higher in rescoring
- Correlation target: `Isolate(k_peak, token_t)` vs `beam_score(token_t)` / CTC probability

### Mechanistic basis
1. **AND-gate features** (high audio dependence) naturally have higher Isolate scores — their activation requires joint presence of audio + context
2. At gc-peak, the model commits to the "Listen" mode — features with high Isolate are the ones *actually driving the decision from audio*
3. Entangled features (low Isolate) = the model is mixing audio + text-prediction signals = ambiguity

### Connection to FAD bias
- Text-predictable phonemes (FAD-biased) have low Isolate at gc-peak (they're being driven by LM context, not audio)
- Rescoring by Isolate would naturally **down-weight** text-predicted tokens and **up-weight** audio-driven ones
- This is a zero-cost interpretability signal for beam search rescoring

## CPU Mock

```python
# isolate_beam_rescore_mock.py
import numpy as np

np.random.seed(42)
N_TOKENS = 100
N_BEAMS = 4

# Mock: gc_peak per token (layer 3-6 for typical Whisper-base)
gc_peak_layer = np.random.randint(3, 7, size=N_TOKENS)

# Mock: Isolate(k_peak) per token — higher = more audio-grounded
# Simulate: correct tokens have higher Isolate, errors have lower
ground_truth_correct = np.random.rand(N_TOKENS) > 0.3  # 70% correct

isolate_signal = np.where(
    ground_truth_correct,
    np.random.uniform(0.5, 1.0, N_TOKENS),   # correct → high Isolate
    np.random.uniform(0.1, 0.5, N_TOKENS)    # error → low Isolate
)

# Mock beam scores (log-prob, higher = better)
beam_scores = np.where(
    ground_truth_correct,
    np.random.uniform(-2.0, -0.5, N_TOKENS),  # correct → less negative
    np.random.uniform(-5.0, -2.0, N_TOKENS)   # error → more negative
)

from scipy import stats
r, p = stats.pearsonr(isolate_signal, beam_scores)
print(f"Pearson r(Isolate, beam_score) = {r:.3f}, p = {p:.2e}")
# Expected: r > 0.6

# Rescoring: rerank beam candidates using Isolate as additional signal
alpha = 0.3  # Isolate weight
rescored = beam_scores + alpha * isolate_signal
r2, p2 = stats.pearsonr(rescored, beam_scores)
print(f"Rescore correlation with original scores: r = {r2:.3f}")

# WER proxy: fraction of low-Isolate tokens selected as best beam
threshold = 0.35
low_isolate_selected = np.mean(isolate_signal < threshold)
print(f"Low-Isolate token fraction: {low_isolate_selected:.2%}")
# Expected: ~30% of tokens are low-Isolate (error-prone)
```

### Expected results:
- `r(Isolate, beam_score) > 0.6` → Isolate is a valid beam quality proxy
- Rescoring with `alpha=0.3` Isolate weight → should improve token-level accuracy
- ~30% of tokens have low Isolate → these are the intervention targets
