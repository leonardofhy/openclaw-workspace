## Core Hypothesis

**Safety training (RLHF/RLAIF) systematically constructs AND-gate features at gc(k) peak layers,
creating a coordination checkpoint that requires audio to "earn" decoder access.**

Corollary: OBLITERATUS-style uncensoring destroys these AND-gate features, reverting the
model to OR-gate (context-sufficient) behavior — explaining why uncensored models become
"audio-agnostic" on safety-critical prompts.

---

## Background

### OBLITERATUS (2025)
A tool that removes safety alignment from open-weight LLMs via weight editing. The key
mechanistic claim (inferred): safety training installs *features* at specific layers that
gate harmful outputs. OBLITERATUS locates and ablates these features.

Our reframing: the "gating features" are AND-gate SAE features at the gc(k) peak layer —
they require BOTH audio signal AND safety context to be present before the decoder proceeds.

### AND-gate Definition (from Q080, Q085, Q091 design docs)
- **AND-gate feature**: SAE feature that fires only when BOTH audio signal AND linguistic
  context are present. Deactivates under noisy audio even if context is intact.
- **OR-gate feature**: SAE feature that fires if EITHER audio OR context is present.
  Does NOT deactivate under noise — context alone activates it.
- **Safety AND-gate**: AND-gate feature at gc(k) peak where the "second input" is a
  safety-relevant context signal (instruction to avoid harm, ethical constraint, etc.)

### gc(k) Coordination Focal Point (from Q102 design doc)
The gc(k) peak layer is the Schelling focal point where audio and language prior "coordinate."
Safety training shifts or strengthens this focal point to also coordinate with safety signals.

## Mechanistic Theory

### Pre-safety training (base model)
```
Audio encoder → [OR-gate features at k*] → Decoder
```
- k* is gc(k) peak for normal audio
- OR-gate: audio OR context → decoder gets signal
- No safety coordination → harmful requests follow high gc(k) path

### Post-safety training (aligned model)
```
Audio encoder → [AND-gate features at k*] → Decoder
                    ↑
              Safety context signal
```
- Safety training modifies OR-gate features → AND-gate features
- Now requires: audio + safety context → decoder proceeds normally
- Without safety context (harmful instruction) → AND-gate BLOCKS → refusal
- gc(k) INCREASES for normal audio (AND-gate is more selective → cleaner signal)
- gc(k) DECREASES for harmful audio (AND-gate blocks transmission → decoder relies on priors)

### OBLITERATUS attack (uncensored model)
```
Audio encoder → [OR-gate features at k*] → Decoder
                    (AND → OR conversion)
```
- OBLITERATUS ablates the safety AND-gate features
- Reverts to OR-gate behavior: harmful audio passes through without check
- gc(k) on harmful prompts RESTORES to baseline (audio signal flows freely again)
- This is the mechanistic explanation of why OBLITERATUS works

## Predictions (Testable Mock Protocol)

### P1: Safety training increases AND-gate fraction at gc(k) peak
```
fAND(k*, aligned) > fAND(k*, base)
fOR(k*, aligned) < fOR(k*, base)
```
Mock: base model has fAND(k*) = 0.3. Aligned model has fAND(k*) = 0.6.
OBLITERATUS model has fAND(k*) restored to ≈ 0.35 (not quite base, some AND-gates remain).

### P2: gc(k) on safety-relevant prompts differs between aligned vs OBLITERATUS
```
gc(k*, aligned, harmful_prompt) << gc(k*, OBLITERATUS, harmful_prompt)
gc(k*, aligned, safe_prompt) ≈ gc(k*, OBLITERATUS, safe_prompt)
```
Key distinction: safety AND-gate is CONDITIONAL on prompt type.
- Safe prompts: aligned ≈ OBLITERATUS (AND-gate fires normally)
- Harmful prompts: aligned has LOW gc (AND-gate blocks), OBLITERATUS has HIGH gc

### P3: gc_peakiness (Q102 metric) predicts refusal rate
```
Pearson r(gc_peakiness_harmful, refusal_rate) < 0
```
Higher gc_peakiness on harmful prompts → model is more audio-responsive to harm → LOWER refusal rate.
Aligned model: low gc_peakiness on harmful = high refusal.
OBLITERATUS: high gc_peakiness on harmful = low refusal.

### P4: OBLITERATUS restores gc(k) decay profile
The aligned model shows faster gc(k) decay after k* for harmful prompts (AND-gate fires briefly, then cuts off).
OBLITERATUS shows sustained gc(k) plateau (OR-gate sustains signal throughout decoding).
Metric: gc_peakiness = gc(k*) / mean(gc) — should be HIGHER for aligned (sharp peak), LOWER for OBLITERATUS.

## "gc(k) Lie Detector" Connection (Q103)

Q103 defines: high gc + refused = "knows-but-refuses" (aligned behavior)
Q097 explains WHY: high gc(k) in aligned model on benign = AND-gate functioning
When model refuses despite audio: AND-gate has BLOCKED the signal (safety gate fired)
So: refusal + high gc before connector = audio was processed, safety gate stopped it downstream
Refusal + low gc = model genuinely didn't process audio (capacity failure)

**This is the mechanistic grounding for Q103's diagnostic protocol.**
