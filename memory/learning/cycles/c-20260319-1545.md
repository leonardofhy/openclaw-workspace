## Problem

We have three independently-derived metrics in the "Listen vs Guess" framework, all purporting to detect when Whisper switches from context-copying to genuine acoustic computation:

1. **gc(k)** — causal patching score at layer k (our primary metric, T3 core)
2. **GSAE graph density** — density of edges in the Graph Sparse Autoencoder's inter-feature graph at layer k
3. **Cascade degree** — mean degree of feature nodes in GSAE (how many downstream features each feature activates)

**Hypothesis**: All three converge to the same latent variable: *the model's reliance on feature-level computation vs. residual copying*. If true, they should all share the same critical point t*, and we can unify them into one GCBench metric.

---

## Method: Theoretical Unification

### Cascade degree → Graph density equivalence

In an undirected GSAE, graph density D(k) = 2|E(k)| / (|V(k)| × (|V(k)|-1)).

Cascade degree c_i = degree of node i = # edges incident to i.

Mean cascade degree = 2|E|/|V| = D(k) × (|V|-1).

**Therefore**: cascade degree and graph density are linearly related (given fixed |V|). They're the *same metric* at different granularities — cascade degree is node-level, graph density is layer-level.

### GSAE graph density → gc(k) equivalence

gc(k) measures causal information flow through specific pathways (patching). GSAE graph density measures the *capacity* of the feature interaction graph.

When the model is context-copying (k < t*): features are sparse and isolated (low connectivity → low density → low cascade degree). The feature graph is thin — few inter-feature dependencies because the model is just relaying context tokens.

When the model is computing acoustics (k ≥ t*): features co-activate in structured clusters (semantic-phonetic entanglement → high connectivity → high density). gc(k) rises because these clustered features *do* carry causal information about the true acoustic target.

**Prediction**: t*(gc) ≈ argmax(graph density(k)) ≈ argmax(mean cascade degree(k))

### Direction of correlation

- gc(k) **rises** at t* (more causal patching = more genuine computation)
- Graph density **rises** at t* (feature clusters forming = genuine computation beginning)
- Both should drop after a late-layer peak as features become task-specific and sparse again

t* is the crossing point. We define it as argmax for gc(k) and as argmax for density.

---

## Three-Metric Unification

| Metric | Level | Direction at t* | Computation |
|--------|-------|-----------------|-------------|
| gc(k) | Layer | ↑ peak | Causal patching (intervention) |
| GSAE graph density D(k) | Layer | ↑ peak | Edge count / max edges |
| Mean cascade degree c̄(k) | Layer | ↑ peak | 2|E|/|V| |

All three are monotone functions of each other (given fixed |V|). The unification claim: **they operationalize the same phase transition**.

---

## GCBench Metric #9

GCBench (Graph Circuit Benchmark) proposes canonical metrics for evaluating graph-based circuit interpretability. Metric #9 should capture the *phase transition quality* of the graph structure.

**Proposed definition**:
```
GCBench-9 = ρ(D(k), gc(k))   [Pearson correlation across layers k=0..K]
```

Where D(k) = GSAE graph density at layer k, gc(k) = causal patching score.

A high GCBench-9 (close to 1.0) means the structural proxy (graph density) is a reliable substitute for the expensive causal patching computation. This matters because:
- gc(k) requires O(n_samples × n_layers) forward passes (expensive)
- D(k) is computable from SAE activations alone (cheap)

If ρ ≈ 0.9+, we can use graph density as a *fast proxy* for gc(k) without running patching experiments.
