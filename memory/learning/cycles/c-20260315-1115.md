## Problem

`gc(k)` measures how much audio information survives to layer k during decoding.
The collapse onset `t*` (Q085) is the decoder step where audio info stops contributing.
**Problem**: By the time t* is detected, it's already "too late" — the generation is already locked into a text-only path.

**Gap**: Can we *predict* t* before it happens? Is there an early warning signal?

---

## Method: T-SAE × gc-incrimination

### T-SAE (Temporal SAE)
- Train (or derive) a Sparse Autoencoder on **temporal activation differences**: `Δh[t] = h[t] - h[t-1]` at a fixed layer across decoding steps
- Alternative: analyze existing SAE feature activations as a time series across decoder steps 1..T
- T-SAE features = dimensions that capture *changes* in the residual stream over time

### gc-incrimination
- Standard causal patching: for each SAE feature `f`, patch its activation at step t and measure Δgc(k)
- "Incriminated features" = high |Δgc(k)| per unit activation change
- Previously used to identify *which features carry audio signal*

### Cross-correlation hypothesis
- Incriminated T-SAE features should **deactivate monotonically** starting at some t_warn < t*
- Rationale: AND-gate features (require both audio + text context) begin to drop out before full collapse, because audio integration becomes increasingly insufficient
- The dropout trajectory of these features IS the early warning signal

---

## H1: Bottleneck Collapse Precursor

```
AND-gate features → start deactivating at t_warn
                        ↓
    gc(k) curve begins declining from peak
                        ↓
    t* = full collapse (gc(k) ≈ 0)

Lead time: t* - t_warn = "warning window"
```

Prediction: t_warn exists, is consistent across samples, and is measurable from SAE activations alone (no ground truth needed).

## Open Questions

1. **T-SAE implementation path**: Train new SAE on Δactivations, OR just track existing SAE feature values as time series? The latter is Tier-1 (CPU, no training). Start there.
2. **Contrastive design**: high-gc samples vs low-gc samples at same text, OR single sample across time? Both valid; the intra-sample temporal design is cleaner.
3. **Lead time consistency**: Is t* - t_warn constant (structural) or variable (content-dependent)? If constant → strong mechanistic claim.
4. **Layer choice**: Which layer to analyze? M7 (Q081 says M7 is the gc peak layer)? Or scan multiple layers?
