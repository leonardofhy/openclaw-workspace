## Task
Q120: Design doc — ENV-type maps to GSAE cluster role; pruning strategy per ENV type

## Conceptual Synthesis

### Framework A: ENV taxonomy (Zhao et al. ESN)
- **ENV-1** (AND-gate): fires only when multiple input features co-active simultaneously
  - Properties: high input-selectivity, non-additive amplification signature
  - In circuit terms: computes a conjunction → acts as a **bottleneck / hub**
- **ENV-3** (OR-gate): fires from any single input, additive
  - Properties: low selectivity, redundant coding, can be reached many ways
  - In circuit terms: computes a disjunction → acts as a **parallel backup / satellite**
- **ENV-2** (mixed): partial conjunction, threshold behavior

### Framework B: GSAE topology
- Features = nodes in co-activation graph
- Edges = pairwise co-activation above threshold
- **Hub node**: high degree, many co-activating partners → many paths pass through it
- **Isolated cluster**: low degree, few connections → satellite node, reached from few paths
- **Bridge node**: connects two communities → often ENV-2 territory?

### Key Mapping

| ENV type | Graph role | Structural signature | Intervention |
|----------|------------|---------------------|-------------|
| ENV-1 (AND-gate) | **Hub node** | High in-degree from diverse neighbors; bottleneck in shortest paths | Reduce hub's edge weights (dampen, not prune — pruning disconnects computation) |
| ENV-3 (OR-gate) | **Isolated cluster node** | Low degree; reachable from few neighbors; high clustering within cluster | Prune entire isolated cluster (loss is recoverable via other OR-paths) |
| ENV-2 (partial) | **Bridge / connector** | Moderate degree; connects different communities | Targeted weight reduction on bridging edges |

### Why this mapping is mechanistically plausible

**AND-gate → hub**: If a feature fires ONLY when multiple inputs co-activate, then it will be co-active with MANY features (all the inputs that must converge). This inflates its degree in the co-activation graph → hub.

**OR-gate → isolated cluster**: If a feature fires from ANY single input, individual activation events are sparse — each triggering input is different. Features that are OR-gates over disjoint inputs form isolated islands in the co-activation graph. They don't reliably co-activate with each other because their triggers are different.

### Intervention Protocol (per ENV type)

**ENV-1 (hub) pruning strategy:**
- Do NOT prune hub node entirely → catastrophic failure (all paths through hub broken)
- Instead: weight-dampen incoming edges with lowest selectivity
- Target: reduce the "OR-gateway" inputs to AND-gate features → make conjunction stricter
- Effect: increases precision (fewer false-positive activations)

**ENV-3 (isolated cluster) pruning strategy:**
- CAN prune entire isolated cluster safely
- Reason: OR-gate features = redundant coding. Other OR-paths compensate.
- This is the jailbreak intervention hypothesis (Q128): jailbreaks activate ENV-3 isolated clusters → prune them → t* restores
- Effect: reduces model's ability to be "fooled" by single-input triggers

**ENV-2 (bridge) strategy:**
- Reduce bridge edge weight selectively
- Monitor community isolation metric pre/post intervention
