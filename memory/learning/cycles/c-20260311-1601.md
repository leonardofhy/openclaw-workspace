## M9: Directed Isolate Metric — Formal Spec for Paper B §3

### Motivation

Current audio SAE evaluation conflates two different types of feature contamination:
- **Output-direction leakage** (RAVEL Isolate): "If I patch feature F to change concept A, does concept B also change?" — tests whether a feature encodes only one concept
- **Input-direction leakage** (NOT in RAVEL): "If concept B is present in the input stimulus, does feature F activate even when concept A is absent?" — tests whether spurious stimulus properties trigger the feature

SPIRIT safety features have HIGH input-direction leakage (angry prosody → safety detector fires on benign content), but current RAVEL Isolate wouldn't catch this because RAVEL Isolate measures output-direction leakage. The two metrics answer different questions about feature quality.

### Definition: Directed Isolate (dIsolate)

Let:
- **F** = feature under test (e.g., safety SAE feature)
- **A** = primary concept (e.g., adversarial jailbreak intent)
- **B** = spurious concept (e.g., angry prosody)

**Existing metrics:**
- `Cause(F, A)` = AUC of F activation to classify A=1 vs A=0 (stimulus control)
- `Isolate_out(F, A)` = 1 - |ΔACT(F) from patching A→B| (RAVEL style, output-direction)

**New metric M9:**
- `Isolate_in(F, A, B)` = 1 - (leakage ratio)

Where **leakage ratio** = `mean_activation(F | B=1, A=0)` / `mean_activation(F | B=0, A=0)`

- Ratio = 1.0 → no input-direction leakage (B doesn't trigger F when A absent)
- Ratio > 1.0 → input-direction leakage (B causes F to fire spuriously)
- `Isolate_in` = 1 − max(0, ratio − 1) clipped to [0, 1]

### Stimulus Protocol (4 conditions, Tier 0 mock-able)

| Condition | A (jailbreak) | B (angry prosody) | Expected F_safety |
|-----------|--------------|-------------------|-------------------|
| C1: double positive | 1 | 1 | HIGH (legitimate) |
| C2: A only | 1 | 0 | HIGH (correct cause) |
| C3: B only | 0 | 1 | SHOULD BE LOW (but ISN'T for SPIRIT) |
| C4: double negative | 0 | 0 | LOW (baseline) |

**Isolate_in** = 1 − (`mean(F | C3)` / `mean(F | C4)` − 1)

If `mean(F | C3) >> mean(F | C4)` → high input leakage → low Isolate_in score.

### Comparison Table (for Paper B §3)

| Metric | Direction | Measures | Failure Mode Caught |
|--------|-----------|----------|---------------------|
| Cause(F, A) | Input | Does A trigger F? | Miss: F low sensitivity |
| Isolate_out(F, A) | Output | Does patching F spill to B? | Miss: input-direction leakage |
| **Isolate_in(F, A, B)** | **Input** | **Does B trigger F spuriously?** | **Miss: none for this type** |

Together: Cause + Isolate_out + Isolate_in = full characterization of feature quality.

### Why This Is Paper B Material (not just a note)

- Gap: No audio SAE benchmark separates input vs output direction leakage
- Novelty: 4/5 — this distinction exists in text interpretability (activation patching direction matters in Elhage et al.) but has NOT been formalized for audio feature evaluation
- Feasibility: 5/5 — Tier 0 mock script: generate 4-condition activation tensors, compute ratio, no model download needed
- **Contribution class**: New metric definition + motivating experiment (SPIRIT re-evaluation under dA-RAVEL)
