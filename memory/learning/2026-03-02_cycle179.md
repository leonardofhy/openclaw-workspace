**ACL 2024 | arXiv:2402.17700 | Stanford + Tel Aviv University + Pr(Ai)²R Group**

## 1. Core Problem: Disentanglement as a Benchmark

RAVEL addresses a precise and underserved question in MI: **given that neurons are polysemantic, can interpretability methods find representations that cleanly isolate ONE attribute from others?**

This is different from "does this feature exist?" — it's "does this feature isolate what you think it isolates, without bleeding into other concepts?"

**The RAVEL disentanglement task:**
- Entity E has multiple attributes A₁, A₂, ... Aₖ (e.g., city Paris → continent, language, timezone, latitude)
- Task: find feature F that encodes attribute Aᵢ **separately** from all other attributes
- Evaluation: if you patch F to value from E' (different entity), does the model predict Aᵢ(E') while keeping all other attributes unchanged?

**Why this is hard:**
- "Paris is in the continent of" → model knows continent = Europe AND language = French AND timezone = CET
- If you patch a "continent" feature, it should output "Asia" (if source = Tokyo), but STILL output "French" (not "Japanese")
- A method fails if patching "continent" also silently changes "language"

## 2. Dataset Structure

Five entity types × multiple attributes each:
| Entity Type | Sample Attributes | # Entities |
|-------------|------------------|------------|
| City | Country, Language, Latitude, Longitude, Timezone, Continent | 3,552 |
| Nobel Laureate | Award Year, Birth Year, Country of Birth, Field, Gender | 928 |
| Verb | Definition, Past Tense, Pronunciation, Singular | 986 |
| Physical Object | Biological Category, Color, Size, Texture | 563 |
| Occupation | Duty, Gender Bias, Industry, Work Location | 799 |

50-150 prompt templates per entity type; zero- and few-shot; JSON format included.

Two evaluation settings:
- **Entity split**: same prompts, unseen entities (tests generalization to new instances)
- **Context split**: same entities, unseen prompt templates (tests generalization to surface form)

**Critical design choice**: Filter to only instances where the LM predicts correctly. This is NOT about finding deficits — it's about understanding why the model SUCCEEDS.

## 3. Evaluation Metric: Interchange Interventions

**Formal definition:**
II(ℳ, F, x, x') = τ(ℳ_{F←GetFeature(ℳ(x'),F)}(x))

Translation: run model on x, but replace feature F with the value F would have taken for input x'. Does the output change to match x's attribute-from-x'?

**Two-score metric:**
- **Cause(A, F)**: did patching F change attribute A's value as expected? (measures F encodes A)
- **Isolate(A, F)**: did patching F leave OTHER attributes unchanged? (measures F isolates A from entangled attributes)

**Overall score**: harmonic mean of Cause + Isolate = RAVEL score

This is the cleanest formalization of "does this feature actually do what you claim?" that I've seen in the field.

---

## 4. Methods Evaluated

RAVEL benchmarks 5 interpretability methods, all cast into the same (N, ℱ, F) framework:
1. **Supervised probes** (linear regression) — standard baseline
2. **PCA** (first principal component of difference activations)
3. **Differential Binary Masking (DBM)** — learns binary masks over neurons
4. **Sparse Autoencoders (SAEs)** — Bricken/Cunningham style
5. **Distributed Alignment Search (DAS)** — rotation-based distributed feature finding
6. **Multi-task DAS (MDAS)** — their new contribution: optimize DAS for multiple attributes simultaneously

**Key results:**
- SAEs are competitive but not SOTA — they score well on Cause but often leak into other attributes (Isolate score drops)
- DAS > SAEs > probes > PCA > random
- **MDAS (multi-task DAS) = SOTA** — optimizing for multiple attributes simultaneously forces proper disentanglement
- SAEs trained with monosemanticity objective (Bricken 2023) still exhibit CROSS-ATTRIBUTE LEAKAGE

**Critical finding for Paper B:**
> SAEs with standard training objectives fail at disentanglement. The RAVEL score exposes leakage that standard reconstruction loss or activation recovery metrics miss.

## 5. Why Probe/PCA Fails at Disentanglement

This is subtle and important. Probes can achieve high Cause scores (accurately predict attribute A) but low Isolate scores (also accidentally encode attribute B). This happens because:
- Probe finds linear direction in activation space that correlates with A
- If A and B are correlated in the data (city's continent correlates with language), that direction also encodes B
- Patching along that direction changes BOTH A and B → disentanglement failure

The SAE analogy: if an audio SAE feature for "voicing" also fires on "speaker gender" (because voiced sounds correlate with male speakers in training data), then patching the voicing feature will also change gender attribution. RAVEL would catch this; standard activation recovery metrics would not.

---

## 6. MDAS — The New Method

Multi-task Distributed Alignment Search:
- Standard DAS: find a rotation R such that R·activations localizes attribute Aᵢ to a subspace
- MDAS: simultaneously find R that localizes ALL attributes to orthogonal subspaces
- Training objective: multi-task causal loss → patch attribute A, don't change attributes B, C, D
- Result: forces representation to be disentangled, not just localized

**Connection to audio**: MDAS is essentially "supervised disentanglement via causal patching objectives". This is a valid method for Gap #18 (does phonological structure survive the connector?) — MDAS could find the rotation that separates voicing from manner from place in the LLM's residual stream after the connector.

**For AudioSAEBench**: Could use MDAS as an upper-bound baseline — "what's the best disentanglement achievable if you optimize directly for it?" and compare SAEs against MDAS as ceiling.

## 7. Key Connections to AudioSAEBench (Paper B)

### RAVEL's four desiderata for interpretability methods:
1. **Faithful** — accurately represents the model
2. **Causal** — identifies components that causally affect behavior
3. **Generalizable** — works on unseen entities/contexts
4. **Isolating** — disentangles individual concepts from entangled representations

**These four desiderata = the design philosophy AudioSAEBench should adopt.**

AudioSAEBench current design:
- Cause score (Gap #22) → ✅ planned (causal consistency via patching)
- Isolation score → ❌ NOT YET PLANNED — this is a gap in our design!
- Generalizability → partially addressed (cross-seed consistency across training runs, but not cross-context in the RAVEL sense)
- Faithfulness → implicit (we test on model's correct predictions via AudioSAE's own probe setup)

**NEW GAP for Paper B**: AudioSAEBench needs an **isolation metric** — does an audio SAE feature for "voicing" leak into "speaker identity" or "noise level"? Need to design an audio version of the Cause/Isolate two-score metric.

**Concrete design for Audio-RAVEL category:**
- Entity type: spoken phoneme pairs (audio stimuli)
- Attributes per entity: voicing, manner, place of articulation, speaker gender, speaking rate, SNR
- Attribute prompt: "What is the phoneme?" (attributes the model reports)
- Cause test: patch voicing feature → does model output change voicing? (e.g., /b/ → /p/)
- Isolate test: patch voicing feature → does model keep manner unchanged? (plosive stays plosive)
- Score: RAVEL-Audio = harmonic mean of Cause + Isolate per feature

**This is new**. RAVEL has no audio version. AudioSAE measures consistency across seeds but not isolation across co-occurring attributes.

## 8. Related Work Connections (for Paper B writing)

RAVEL positions against:
- Probing literature (Peters 2018, Hupkes 2018, Tenney 2019) — measures what's decodable, not what's causal
- Activation patching (Geiger, Meng, Wang, etc.) — measures causal necessity, not isolation
- SAE literature (Bricken 2023, Cunningham 2024) — measures reconstruction quality, not disentanglement quality
- DAS (Geiger 2023b, Wu 2023) — measures causal localizability, not multi-attribute isolation

**AudioSAEBench positioning**: "RAVEL established disentanglement evaluation for text LMs; AudioSAEBench extends this to speech models, replacing entity-attribute pairs with phonological stimuli and introducing audio-specific metrics (TCS(F), temporal causal consistency)."

This is a clean, defensible related work structure.

## 9. Audio-Specific Extension Challenges

Three challenges for AudioRavel (not in RAVEL's scope):
1. **No discrete attribute vocabulary**: text entity attributes have clean values (Continent = "Europe" | "Asia" | ...). Audio phonological attributes are continuous (voicing degree, formant frequency). Need to binarize (voiced/unvoiced) or use graded metric.
2. **Sequentiality**: audio features change over time within one stimulus. Need to define "the voicing feature" at which time step?
3. **Minimal pair availability**: RAVEL has 3500+ city entities. Validated minimal pair stimuli are scarce (Choi et al. uses ~96 languages × ~10 contrasts = limited).

**Solutions** (draft ideas for Paper B):
1. Use binary phonological features (voicing, nasality, plosive/fricative) — well-defined from phonetics
2. Use frame-level SAE features at the specific frame of the target phoneme (MFA alignments provide ground truth)
3. Use Choi et al. 2602.18899 stimuli + augment with TTS-generated pairs (acknowledge synthetic as secondary)

## 10. Open Questions Generated

1. **Can MDAS be applied to Whisper?** MDAS requires differentiable activation editing — pyvene supports this. If yes, MDAS is a natural ceiling baseline for AudioSAEBench.

2. **Is the leakage problem WORSE in audio?** Text attributes (continent, language) are correlated because of world knowledge patterns in training data. Audio attributes (voicing, speaker gender) are correlated due to acoustic co-occurrence — arguably MORE entangled at the physical signal level. Audio SAEs might systematically leak more → AudioSAEBench isolation metric would be more discriminative than for text.

3. **Does MDAS generalize to discrete speech tokens?** MDAS works in continuous activation space. For codec-based speech LLMs that use discrete EnCodec tokens, the activation space after embedding might not be well-suited for rotation-based disentanglement. This is a boundary case worth noting.
